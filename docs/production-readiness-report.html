Executive summary

Modern-EHR has a clean architecture, sensible auth split between staff and patient surfaces, structured audit logging, defensive scoping of PHI access, and passing unit tests on the security-critical paths. It is in good shape for a controlled internal pilot or a clinical alpha. However, several items required to operate as a HIPAA-covered production system are not yet in place — they are listed below as Critical and High findings. None of these are large lifts individually, but together they should block a public production launch with real PHI until resolved.

Key flows verified end-to-end

  • Staff login → JWT (token_type=user) → role-gated endpoints (backend/app/api/deps.py:34).
  • Patient invite → setup token (SHA-256 hashed at rest) → patient JWT (token_type=patient) → patient-portal endpoints (deps.py:90).
  • Cross-surface token rejection: patient JWT on /users → 401; staff JWT on /patient-portal/me → 401.
  • Real-time messaging: REST write → ws_manager.send_to_user() → React Query invalidation on both ends.
  • Non-admin scoping: providers see only their own tasks + assigned patients (endpoints/tasks.py:51, endpoints/patients.py:53).

3 · HIPAA Security Rule alignment

This is the most consequential section. HIPAA isn't a checklist you pass with code alone — it requires policies, BAAs, and operational controls. The code-side picture is below.

§164.312(a)(1) — Access controls

  • Unique user identification — UUID per user/patient; JWTs scoped by sub.
  • Role-based authorisation — provider / staff / admin enforced server-side via require_roles().
  • Emergency access procedure — no documented break-glass mechanism. Need a documented admin-override audit pattern.
  • Automatic logoff (164.312(a)(2)(iii)) — access tokens last 60 min; refresh extends 14 days. Best practice is 15 min idle with hard refresh rotation. High
  • Encryption / decryption of stored PHI — Postgres column-level encryption is not used; relies on disk-level / managed-DB encryption from the host (e.g. RDS encryption). Document this as part of the architecture.

§164.312(b) — Audit controls

  • audit_logs table captures user_id, action, resource_type, resource_id, ip_address, user_agent, payload, timestamp (models/audit_log.py).
  • 78 audit-write call sites across the v1 router — every mutating endpoint plus auth events records to audit_logs.
  • Read-access audit is partial. PHI reads of /patients/{id} are audited, but list reads, dashboard composition reads, and patient-portal reads are not. HIPAA mainly requires that you can reconstruct who saw what; consider a request-level access log.
  • Payload scrubbing — payload JSONB occasionally captures request payloads. There is no central scrubber to mask passwords, tokens, or freeform PHI before persisting. Audit logs must never become a secondary PHI / credential store. High
  • Tamper resistance — logs live in the same database as PHI with no append-only constraint. Best practice: stream a copy to an immutable sink (S3 Object Lock / CloudWatch).

§164.312(c) — Integrity

  • All updates go through ORM models; no raw SQL string concatenation found in services.
  • Foreign-key cascades + check constraints exist in migrations.
  • No row-level integrity checksum or signed-record mechanism. Most clinical reviewers accept relational integrity + audit trail; if your reviewer expects FHIR-style record provenance you'll need to add it.

§164.312(e) — Transmission security

  • TLS termination is delegated to the deployment layer (k8s ingress / ALB / nginx). Code does not configure it. Document the prod topology explicitly.
  • WebSocket auth uses a JWT in the query param; on connect the server decodes and verifies before accept().
  • Patient WebSocket origin — the WS endpoint shares the manager with staff. UUIDs don't collide, but a documented threat model is needed for the shared connection pool.

HIPAA Privacy Rule — "minimum necessary"

  • Patient portal services hide unrelated patients; provider portal scopes /patients to assigned caseload for non-admin.
  • Patient-portal endpoints return narrow projections (PatientMeOut, PatientAppointmentOut etc.), not full models.
  • Admin role currently sees everything in /users, /patients, /tasks. Acceptable for small clinics; large deployments need finer admin sub-roles (audit-admin vs clinic-admin).
HIPAA reality check. Source code is necessary but not sufficient. Production HIPAA compliance requires: a signed BAA with every infra provider (DB, LLM, email, SMS), a documented Security Risk Analysis, a Workforce training programme, a Breach-Notification policy, and signed BAAs with every downstream service. The code-side gaps below are the lift on your side.

8 · Final verdict

Modern-EHR is functionally complete and architecturally sound for a pilot with synthetic or de-identified data. The auth split between staff and patient surfaces is a meaningful security win; the patient-portal write paths defend ownership at the service layer; the audit log captures 78 mutating endpoints; the codebase type-checks clean on both Python and TypeScript sides. The 59 backend tests all pass.

It is not yet ready to operate as a HIPAA-covered production system with real PHI. The four Critical items in the table above (LLM BAA, rate limiting on auth, refresh-token rotation, audit-log payload scrubbing) are blockers in the strict sense — each of them is a small engineering lift, but each is a clear control gap that an OCR audit or any serious payor security review will flag. The High items should be closed during the pilot phase.

Recommended path to production: close items #1-#10 over a 2-3 week hardening sprint, run a controlled pilot with a small set of consented patients (or fully synthetic data), engage a third party for an independent SRA / penetration test, and only then enable real PHI ingestion.
